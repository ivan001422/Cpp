| 17        | `,`                                                                       | Оператор запятая                                               | слева направо   |

## Примечания

- Группа 2 (постфиксные) и группа 3 (унарные) имеют разную ассоциативность: постфиксные читаются слева направо, унарные — справа налево. Это важно, например, в выражении `*p++` — сначала `p++` (постфиксный), затем `*` (унарный).
- Операторы приведения типов (`static_cast`, `dynamic_cast` и др.) и `typeid` находятся на втором уровне.
- В группе 16 перечислены основные составные присваивания; их ассоциативность — справа налево, что позволяет цепочки `a = b = c`.
- Тернарный оператор (15) также правоассоциативен, что позволяет вложенные конструкции `a ? b : c ? d : e`.
- Оператор запятая имеет самый низкий приоритет, поэтому выражения вроде `int x = 1, y = 2;` работают как ожидается.

## Другие операции

##  Динамическая идентификация типов (RTTI)

**RTTI (Runtime Type Identification)** — механизм, позволяющий узнать точный тип объекта во время выполнения. Актуален для **полиморфных** иерархий (с виртуальными функциями), где указатель на базовый класс может ссылаться на объект производного класса.

### Операция `typeid`

Возвращает объект типа `type_info`, содержащий информацию о типе.

**Заголовок:** `<typeinfo>`

**Синтаксис:**
```cpp
typeid(объект)      // для объекта или выражения
typeid(имя_типа)    // для типа
```

**Класс `type_info`** (основные методы):
- `name()` — возвращает строку с именем типа (зависит от компилятора).
- `operator==` и `operator!=` — сравнение типов.

**Пример:**
```cpp
if (typeid(i) == typeid(j)) { /* типы совпадают */ }
cout << typeid(MyClass).name(); // вывод: class MyClass
```

**Важное поведение:**
- Если `typeid` применяется к **полиморфному** базовому классу (с виртуальной функцией), он возвращает **реальный тип** объекта, на который указывает указатель.
- Для **неполиморфного** класса возвращается тип указателя (базовый), а не реальный объект.
- При применении к нулевому указателю генерируется исключение `bad_typeid`.

## Операции приведения типов (casting)

В C++ есть 5 операций приведения. Четыре из них (`dynamic_cast`, `const_cast`, `static_cast`, `reinterpret_cast`) дают более тонкий контроль, чем традиционный `(тип)выражение`.

| Операция | Назначение | Проверка во время выполнения |
| :--- | :--- | :--- |
| **`dynamic_cast`** | Безопасное приведение в иерархиях классов (полиморфных) | **Да** (если нельзя, возвращает 0 для указателей или исключение `bad_cast` для ссылок) |
| **`const_cast`** | Добавление/удаление `const` или `volatile` | Нет |
| **`static_cast`** | Все стандартные (неполиморфные) преобразования | Нет |
| **`reinterpret_cast`** | Опасное приведение между несвязанными типами (битовое копирование) | Нет |
| Традиционное `(тип)` | Устаревший вариант, заменяется на `static_cast` и др. | Нет |

### `dynamic_cast` (самый важный)
```cpp
Base *bp = new Derived();
Derived *dp = dynamic_cast<Derived*>(bp);
if (dp) { /* приведение успешно */ }
```

Позволяет безопасно привести указатель на базовый класс к указателю на производный, если объект действительно является производным.

### `const_cast`
```cpp
const int a = 5;
int *p = const_cast<int*>(&a); // удаляем const (ОПАСНО!)
```

Меняет константность. Используйте только если уверены, что объект не является константным по факту.

### `static_cast`
```cpp
double d = 3.14;
int i = static_cast<int>(d); // стандартное преобразование
```
Выполняет обычные преобразования (int->double, указатели в иерархии без проверок, и т.д.).

# 📘 Ручное управление памятью в C++: `new` и `delete`

В C++ память можно выделять не только на стеке (автоматически), но и в **куче (heap)** — области памяти, которая существует до явного освобождения. Для этого используются операторы `new` и `delete`. Они дают полный контроль над временем жизни объектов, но требуют дисциплины, чтобы избежать ошибок. В этом конспекте мы разберём, как они работают, в чём их опасность и какие современные альтернативы предлагает C++.

---

## Зачем нужно ручное управление памятью?

- Стек ограничен по размеру (обычно несколько мегабайт). Куча позволяет выделять большие объёмы.
- Объекты, созданные в куче, живут до явного удаления, а не до конца области видимости.
- Это нужно для динамических структур данных, когда размер заранее неизвестен (например, связные списки, деревья).

Однако ручное управление чревато ошибками: утечки памяти, двойное удаление, доступ к уже освобождённой памяти. Поэтому в современном C++ прямое использование `new`/`delete` сведено к минимуму — вместо них применяются **умные указатели** и контейнеры. Но понимать механизм необходимо, чтобы читать старый код и осознавать, как работают эти абстракции.

---

## Выделение и освобождение памяти для одного объекта

### `new` – выделение и (опционально) инициализация
```cpp
int* p = new int;        // выделяет память под int, не инициализирует
int* q = new int(5);     // выделяет и инициализирует значением 5
int* r = new int{10};    // uniform-инициализация (C++11)
```
`new` возвращает указатель на выделенную память. Если память не может быть выделена, бросается исключение `std::bad_alloc`.

### `delete` – освобождение памяти
```cpp
delete p;   // освобождает память, на которую указывает p
p = nullptr; // хороший тон: обнулить указатель, чтобы избежать висячего указателя
```

### Важно:
- После `delete` указатель становится **висячим** (dangling) — обращение к нему приводит к неопределённому поведению.
- Удалять `nullptr` безопасно (ничего не происходит).
- Нельзя удалять одну и ту же память дважды (double delete) — это UB.

---

## Выделение массивов: `new[]` и `delete[]`

Для выделения массива объектов используется оператор `new[]`, а для освобождения — `delete[]`.

```cpp
int* arr = new int[10];       // массив из 10 int (не инициализирован)
int* arr2 = new int[10]();    // инициализирован нулями
int* arr3 = new int[10]{1,2,3}; // первые 3 элемента, остальные нули (C++11)

// Использование
arr[0] = 42;

// Освобождение
delete[] arr;   // обязательно с квадратными скобками!
arr = nullptr;
```

**Ошибка:** использование `delete` вместо `delete[]` для массива — неопределённое поведение (обычно приводит к утечке или крашу). Всегда соблюдайте пары: `new` → `delete`, `new[]` → `delete[]`.

---

## Инициализация при выделении

| Синтаксис | Результат |
|-----------|-----------|
| `new int` | не инициализирован (мусор) |
| `new int()` | инициализирован нулём (value-initialization) |
| `new int(5)` | инициализирован 5 |
| `new int{5}` | инициализирован 5 (C++11) |
| `new int[5]` | массив из 5 int, не инициализирован |
| `new int[5]()` | все элементы инициализированы нулём |
| `new int[5]{1,2}` | первые два — 1 и 2, остальные нули (C++11) |

Для классов вызывается соответствующий конструктор (или конструктор по умолчанию).

---

## Основные ошибки при работе с new/delete

### Утечка памяти (memory leak)
```cpp
int* p = new int(10);
p = new int(20); // старый объект потерян, память не освобождена
delete p;       // удаляется только второй
```
Чтобы избежать, всегда освобождайте память перед переприсваиванием.

### Двойное удаление (double delete)
```cpp
int* p = new int(5);
delete p;
delete p; // UB: попытка удалить уже удалённую память
```

### Висячий указатель (dangling pointer)
```cpp
int* p = new int(5);
delete p;
*p = 10; // UB: доступ к освобождённой памяти
```

### Несоответствие `new[]` и `delete` (или наоборот)
```cpp
int* arr = new int[10];
delete arr; // UB — нужно delete[]
```

### Исключения и утечки
Если между `new` и `delete` возникает исключение, память не освободится:
```cpp
int* p = new int(5);
dangerous_function(); // если бросит исключение, delete не выполнится
delete p;
```
Решение — использовать умные указатели, которые освободят память автоматически при раскрутке стека.

---

## Сравнение `new/delete` с `malloc/free` (из C)

В C++ также доступны функции `malloc` и `free` из C, но их использовать не рекомендуется:

| `new` / `delete` | `malloc` / `free` |
|------------------|-------------------|
| Операторы языка | Функции библиотеки |
| Вызывают конструктор / деструктор | Не вызывают |
| Возвращает указатель нужного типа (`int*`) | Возвращает `void*`, нужно приведение |
| Бросает `std::bad_alloc` при ошибке | Возвращает `nullptr` при ошибке |
| Размер вычисляется автоматически | Нужно указывать размер в байтах |
| Поддерживает перегрузку | Нет перегрузки |

**Правило:** в C++ всегда используйте `new`/`delete` (или умные указатели), а не `malloc`/`free`.

## Современные альтернативы: умные указатели и RAII

Начиная с C++11, стандартная библиотека предоставляет **умные указатели**, которые автоматически управляют временем жизни объекта. Они реализуют идиому RAII (Resource Acquisition Is Initialization) — ресурс (память) захватывается в конструкторе и освобождается в деструкторе.

### `std::unique_ptr` – эксклюзивное владение
```cpp
#include <memory>
std::unique_ptr<int> p = std::make_unique<int>(5);
// при выходе из области видимости память будет освобождена
// копирование запрещено, можно перемещать
```

### `std::shared_ptr` – разделяемое владение (счётчик ссылок)
```cpp
std::shared_ptr<int> p = std::make_shared<int>(10);
std::shared_ptr<int> q = p; // оба указывают на один объект
// память освободится, когда последний shared_ptr будет разрушен
```

### `std::weak_ptr` – для предотвращения циклических ссылок (дополнительно)

**Преимущества умных указателей:**
- Автоматическое освобождение памяти (даже при исключениях).
- Явно выражают семантику владения.
- Избавляют от ручного `delete`, сокращая количество ошибок.

Поэтому в современном коде явные `new` и `delete` встречаются редко — только в низкоуровневых библиотеках или при взаимодействии с C-кодом. Вместо них используйте `std::vector`, `std::string` и умные указатели.

---

## Когда всё же приходится использовать `new`/`delete`?

- В старом легаси-коде.
- При написании собственных контейнеров или умных указателей.
- При работе с библиотеками на C, где требуется передача сырых указателей (но часто можно обернуть в умные).
- В некоторых низкоуровневых оптимизациях (например, placement new для размещения в заранее выделенном буфере).

В таких случаях помните о правилах:
- Всегда обнуляйте указатели после `delete`.
- Используйте `std::unique_ptr` для явного управления, если возможно.
- Не смешивайте разные способы управления памятью.

# 📘 Умные указатели в C++

Умные указатели — это классы, которые ведут себя как обычные указатели, но автоматически управляют временем жизни объектов в куче. Они реализуют принцип **RAII (Resource Acquisition Is Initialization)**: память выделяется при создании указателя и освобождается при его уничтожении. Это избавляет от ручного вызова `delete` и предотвращает утечки памяти, двойное освобождение и висячие указатели.

В C++11 стандартная библиотека предоставляет три основных типа умных указателей:

- **`unique_ptr`** – исключительное владение
- **`shared_ptr`** – разделяемое владение
- **`weak_ptr`** – невладеющий наблюдатель

> `auto_ptr` был признан устаревшим в C++11 и удалён в C++17.

---

## Подключение

Все умные указатели определены в заголовочном файле `<memory>`:

```cpp
#include <memory>
using namespace std;
```

---

## `unique_ptr` – уникальное владение

`unique_ptr` — это умный указатель, который **единолично владеет** объектом в куче. Он не может быть скопирован, но может быть перемещён (move semantics). Когда `unique_ptr` уничтожается (например, при выходе из области видимости), он автоматически вызывает `delete` для управляемого объекта.

### Создание

```cpp
// C++11
unique_ptr<int> ptr1(new int(10));

// C++14 и новее (рекомендуемый способ)
auto ptr2 = make_unique<int>(20); // инициализация значением 20
auto ptr3 = make_unique<int[]>(5); // массив из 5 элементов
```

> `make_unique` предпочтительнее, чем `new`, так как он короче, безопаснее при исключениях и выразительнее.

### Основные операции

```cpp
unique_ptr<int> p1 = make_unique<int>(42);

// Доступ к объекту
*p1 = 100;                // разыменование
int val = *p1;

// Перемещение (передача владения)
unique_ptr<int> p2 = move(p1); // p1 становится nullptr
// unique_ptr<int> p3 = p1; // ОШИБКА: копирование запрещено

// Проверка на nullptr
if (p2) { /* указатель не пуст */ }

// Освобождение ресурса вручную
p2.reset();                // удаляет объект, p2 = nullptr
p2.reset(new int(50));     // удаляет старый, захватывает новый

// Открепление владения
int* raw = p2.release();   // p2 = nullptr, память не освобождена
delete raw;                // нужно удалить вручную
```

### `unique_ptr` как член класса

```cpp
class MyClass {
    unique_ptr<Resource> res_;
public:
    MyClass() : res_(make_unique<Resource>()) {}
    // Деструктор не нужен — res_ удалится автоматически
};
```

### Возврат `unique_ptr` из функции

```cpp
unique_ptr<Resource> createResource() {
    return make_unique<Resource>();
}

auto resource = createResource(); // перемещение, эффективно
```

### Пользовательский удалитель

По умолчанию `unique_ptr` использует `delete`. Можно задать свой способ освобождения ресурса:

```cpp
// Для FILE*
auto deleter = [](FILE* f) { if (f) fclose(f); };
unique_ptr<FILE, decltype(deleter)> file(fopen("test.txt", "w"), deleter);
```

### Производительность

`unique_ptr` не имеет накладных расходов по памяти и времени по сравнению с сырым указателем.

---

## `shared_ptr` – разделяемое владение

`shared_ptr` реализует **разделяемое владение**. Несколько `shared_ptr` могут указывать на один и тот же объект. Объект будет уничтожен, когда **последний** `shared_ptr`, указывающий на него, будет разрушен. Это достигается с помощью **счётчика ссылок** (reference count).

### Создание

```cpp
// C++11
shared_ptr<int> ptr1(new int(10));

// C++11 и новее (рекомендуемый способ)
auto ptr2 = make_shared<int>(20);
```

> `make_shared` предпочтительнее: он выполняет одно выделение памяти и для объекта, и для блока управления, что эффективнее.

### Основные операции

```cpp
shared_ptr<int> p1 = make_shared<int>(42);
shared_ptr<int> p2 = p1;          // копирование разрешено

*p1 = 100;                         // изменение через любой указатель

cout << p1.use_count();           // 2 (два указателя)

p2.reset();                        // p2 больше не указывает на объект
cout << p1.use_count();           // 1

// Проверка уникальности владения
if (p1.unique()) { /* только один владелец */ }
```

### Сравнение с raw pointer

```cpp
// Сырой указатель — опасно
{
    int* p = new int(100);
    // ... если здесь исключение — утечка
    delete p;
}

// shared_ptr — безопасно
{
    auto p = make_shared<int>(100);
    // ... даже при исключении память будет освобождена
}
```

### `shared_ptr` для массивов

В C++17 `shared_ptr` не имеет встроенной поддержки массивов, но можно использовать пользовательский удалитель:

```cpp
shared_ptr<int[]> p(new int[10]); // C++17
// или
shared_ptr<int> p(new int[10], default_delete<int[]>());
```

С C++20 `make_shared` поддерживает массивы.

## `weak_ptr` – слабая ссылка

`weak_ptr` — это **невладеющий** наблюдатель, который указывает на объект, управляемый `shared_ptr`. Он **не увеличивает** счётчик ссылок. Это позволяет безопасно наблюдать за объектом, не продлевая его жизнь.

### Основное назначение: предотвращение циклических ссылок

Когда два `shared_ptr` указывают друг на друга, возникает цикл: счётчики ссылок никогда не станут нулевыми, и объекты не будут удалены. `weak_ptr` разрывает такие циклы.

```cpp
struct Node {
    shared_ptr<Node> next;
    weak_ptr<Node> prev; // слабая ссылка на предыдущий узел
};

auto n1 = make_shared<Node>();
auto n2 = make_shared<Node>();
n1->next = n2;
n2->prev = n1; // prev — weak_ptr, цикл не образуется
```

### Использование `weak_ptr`

```cpp
shared_ptr<int> sp = make_shared<int>(42);
weak_ptr<int> wp = sp;    // наблюдение за sp

// Проверка, жив ли объект
if (auto locked = wp.lock()) { // блокировка: создаёт shared_ptr
    cout << *locked;           // объект жив
} else {
    cout << "Объект удалён";
}

// Альтернативная проверка
if (!wp.expired()) {
    // объект ещё существует
}
```

### Важные методы

| Метод | Описание |
|-------|----------|
| `lock()` | Возвращает `shared_ptr` к объекту, если он жив, иначе пустой указатель |
| `expired()` | Возвращает `true`, если объект уже удалён |
| `reset()` | Очищает `weak_ptr` |

## Сравнение умных указателей

| Характеристика | `unique_ptr` | `shared_ptr` | `weak_ptr` |
|----------------|--------------|--------------|------------|
| Владение | Исключительное | Разделяемое | Нет (наблюдатель) |
| Копирование | ❌ запрещено | ✅ разрешено | ✅ разрешено |
| Перемещение | ✅ | ✅ | ✅ |
| Счётчик ссылок | Нет | Есть | Нет (наблюдает за счётчиком) |
| Накладные расходы | Нет | Есть (контрольный блок) | Есть (как у `shared_ptr`) |
| Размер | Как у сырого указателя | Два указателя | Два указателя |
| Применение | Один владелец | Много владельцев | Наблюдение + разрыв циклов |

## Распространённые ошибки и как их избежать

### ❌ Создание `shared_ptr` из сырого указателя

```cpp
int* raw = new int(10);
shared_ptr<int> sp1(raw);
shared_ptr<int> sp2(raw); // ОШИБКА: два независимых счётчика!
```

**Решение:** Всегда создавайте `shared_ptr` через `make_shared` или сразу из `new`.

### ❌ Циклические ссылки с `shared_ptr`

```cpp
struct A { shared_ptr<A> next; };
auto a1 = make_shared<A>();
auto a2 = make_shared<A>();
a1->next = a2;
a2->next = a1; // цикл! память никогда не освободится
```

**Решение:** Используйте `weak_ptr` для одного из направлений.

### ❌ Смешивание умных и сырых указателей

```cpp
auto sp = make_shared<int>(42);
int* raw = sp.get();
delete raw; // ОШИБКА: двойное удаление!
```

**Решение:** Не удаляйте сырые указатели, полученные через `.get()`.

## Рекомендации по использованию

| Ситуация | Рекомендация |
|----------|--------------|
| Требуется один владелец | `unique_ptr` |
| Требуется несколько владельцев | `shared_ptr` |
| Требуется наблюдение без владения | `weak_ptr` |
| Создание указателя | `make_unique` / `make_shared` |
| Возврат из функции | `unique_ptr` |
| Передача в функцию | `const unique_ptr<T>&` или `T*` для невладеющего доступа |
| Массивы | `unique_ptr<T[]>` |

## Заключение

Умные указатели — это фундаментальный инструмент современного C++ для безопасного управления памятью. Они реализуют принцип RAII и избавляют от большинства проблем, связанных с ручным управлением памятью.

**Золотое правило:** в современном C++ код, использующий явные `new` и `delete`, должен быть исключением. Используйте умные указатели и контейнеры STL для автоматического управления ресурсами.
